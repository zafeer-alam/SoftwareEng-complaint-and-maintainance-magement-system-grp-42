    System Flow Diagram

    Campus Complaint & Maintenance Management System

    End-to-end system architecture and data flow

    Complaint Flow
    1. User submits complaint
    2. Backend validates & stores
    3. Admin assigns to staff
    4. Staff updates progress
    5. User notified on resolve
    Architecture
    • 3-Tier REST Architecture
    • JWT Authentication
    • Role-based Access Control
    • Modular backend services
    • MongoDB Atlas DB
    Non-Functional
    • Response time < 2 sec
    • Scalable architecture
    • Encrypted passwords (bcrypt)
    • Netlify / Vercel frontend
    • Render / AWS backend
    API Reference
    🔐 Authentication
    POST/api/auth/registerRegister new user
    POST/api/auth/loginLogin and get JWT token
    POST/api/auth/logoutInvalidate session
    📝 Complaints
    POST/api/complaintsCreate new complaint
    GET/api/complaintsList all complaints (admin)
    GET/api/complaints/:idGet complaint by ID
    PUT/api/complaints/:idUpdate status / details
    DEL/api/complaints/:idDelete complaint (admin)
    GET/api/complaints/user/:userIdComplaints by user
    📌 Assignment
    POST/api/assignAssign complaint to staff
    GET/api/assign/staff/:staffIdAssignments for staff
    PUT/api/assign/:id/statusUpdate assignment status
    📊 Reports
    GET/api/reports/summaryOverall statistics
    GET/api/reports/categoriesBreakdown by category
    GET/api/reports/staff/:idPerformance by staff
    GET/api/reports/monthlyMonthly volume chart
    🔔 Notifications
    GET/api/notifications/:userIdUser notifications
    PUT/api/notifications/:id/readMark as read

    🔗 Relationships
    Users 1 → N Complaints   (one user can file many complaints)
    Complaints 1 → 1 Assignments   (each complaint assigned to one staff)
    Users (staff) 1 → N Assignments   (staff handles many assignments)
    Complaints 1 → N Status_Logs   (full audit trail)
